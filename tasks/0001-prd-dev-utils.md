# Product Requirements Document: Dev Utils

## 1. Introduction / Overview

**Dev Utils**는 개발자의 일상적인 작업 생산성을 향상시키기 위한 통합 도구 모음입니다. JSON 검증, JWT 디코딩, SQL 쿼리 변환 등 자주 사용되는 개발 작업들을 한 곳에서 수행할 수 있게 하며, 특히 민감한 정보를 로컬에서만 처리하여 보안을 강화합니다.

**주요 가치제안**:
- 🔒 모든 데이터 변환을 클라이언트 사이드에서 처리 (보안)
- ⚡ 구글링 없이 자주 사용하는 변환 작업 자동화
- 📋 직관적인 UI로 빠른 작업 수행

## 2. Goals

1. **개발자 생산성 향상**: 자주 사용되는 변환/검증 작업을 3초 이내에 완료
2. **보안 강화**: 민감한 정보(JWT, DB 설정 등)를 외부 서비스 없이 처리
3. **개발 효율성**: 온라인 도구 검색 및 이동 시간 제거
4. **플랫폼 표준화**: Next.js + Shadcn UI로 사내 기술 스택 통일
5. **점진적 확장**: MVP 이후 추가 도구를 단계적으로 추가 가능한 구조

## 3. User Stories

### 3.1 초급 개발자
- **AS A** 신입 개발자
- **I WANT TO** JSON을 쉽게 검증하고 포맷팅하기
- **SO THAT** API 응답을 빠르게 확인할 수 있고 학습할 수 있다

### 3.2 백엔드 개발자
- **AS A** Spring Boot 개발자
- **I WANT TO** 준비된 SQL(?)과 바인딩 파라미터를 완전한 쿼리로 변환하기
- **I WANT TO** 디버깅할 때 더 빠르게 쿼리를 실행할 수 있다

### 3.3 보안 담당자
- **AS A** DevOps/보안 담당자
- **I WANT TO** JWT 토큰을 로컬에서 디코딩하기
- **SO THAT** 민감 정보가 외부 서비스에 노출되지 않는다

### 3.4 팀원
- **AS A** 팀 멤버
- **I WANT TO** 팀에서 자주 쓰는 변환 도구들을 한 곳에서 찾기
- **SO THAT** 생산성이 향상되고 일관된 도구를 사용한다

## 4. Functional Requirements

### MVP 범위 (Phase 1)

#### 4.1 JSON Formatter & Validator
**FR1.1** 사용자가 미포맷팅된 JSON 문자열을 입력하면 들여쓰기를 적용하여 표시한다
- 예: `{"name":"John","age":30}` → 보기 좋게 정렬
- 오류 시 구체적인 에러 메시지 표시

**FR1.2** JSON 문법 오류를 감지하고 위치와 함께 오류 메시지를 표시한다
- 예: "Line 3, Column 5: Expected comma"

**FR1.3** 포맷팅된 JSON을 한 번의 클릭으로 클립보드에 복사한다
- Toast 메시지로 복사 완료 알림

**FR1.4** 원본 JSON과 포맷팅된 JSON을 동시에 표시한다
- 좌측: 입력값, 우측: 포맷팅된 결과

#### 4.2 JWT 디코더 (오프라인)
**FR2.1** 사용자가 JWT 토큰을 입력하면 Header와 Payload를 파싱하여 표시한다
- Base64 디코딩은 브라우저 내에서만 처리

**FR2.2** 토큰의 각 부분(Header, Payload, Signature)을 분리하여 표시한다
- Header: 알고리즘 정보
- Payload: 실제 데이터
- Signature: 서명 값

**FR2.3** Payload의 내용을 JSON 형식으로 보기 좋게 표시한다
- 클레임 정보를 읽기 쉽게 정렬

**FR2.4** 토큰 검증 결과를 표시한다 (형식 검증만)
- 유효한 JWT 형식인지 확인
- 서명 검증은 수행하지 않음 (오프라인 한계)

#### 4.3 SQL 파라미터 바인딩
**FR3.1** 사용자가 PreparedStatement SQL(?)과 바인딩 파라미터를 입력하면 실행 가능한 SQL로 변환한다
- 예: `SELECT * FROM users WHERE id = ?` + `[123]` → `SELECT * FROM users WHERE id = 123`

**FR3.2** 파라미터 개수와 ?의 개수가 일치하지 않으면 오류 메시지를 표시한다
- "파라미터 개수 불일치: 3개의 ?가 있지만 2개의 파라미터만 제공됨"

**FR3.3** 문자열 파라미터를 따옴표로 감싼다
- 예: `[123, 'admin']` → `123, 'admin'`

**FR3.4** 변환된 SQL을 클릭으로 클립보드에 복사한다

### 향후 기능 (Phase 2+)

#### 추가 예정 도구들
- YAML ↔ Properties 변환기
- Epoch/Unix Timestamp 변환기
- Cron Expression 생성기
- URL Encode/Decode
- Redis Key 패턴 스캐너
- 에러 코드 조회기
- RegEx 테스트 도구

## 5. Non-Goals (Out of Scope)

- **서버 사이드 처리**: 모든 변환은 클라이언트에서만 처리 (보안)
- **데이터베이스 연결**: 실제 DB 쿼리 실행 기능 없음
- **사용자 인증**: 로그인 시스템 불필요
- **데이터 저장**: 변환 기록이나 즐겨찾기 기능 없음 (MVP 범위)
- **API 통합**: 외부 서비스 연동 없음
- **Signature 검증**: JWT 서명 검증은 수행하지 않음

## 6. Design Considerations

### 6.1 UI/UX 구조
- **Tabs 컴포넌트**: 상단에 여러 도구를 탭으로 구분
  - [JSON] | [JWT] | [SQL] | ... 형태

- **레이아웃**:
  - 좌측: 입력 영역 (Textarea)
  - 우측: 결과 영역 (읽기 전용 또는 선택 가능)
  - 하단: 복사, 초기화 버튼

### 6.2 Shadcn UI 컴포넌트 활용
- `Tabs`: 도구 선택
- `Textarea`: 입력 필드
- `Card`: 결과 표시
- `Button`: 액션 (복사, 초기화, 변환)
- `Toast` (Sonner): 복사 완료, 에러 메시지 알림
- `Alert`: 오류 표시

### 6.3 시각적 피드백
- 에러 상태: 빨간색 테두리 + 에러 메시지
- 성공 상태: 초록색 체크마크 + Toast 알림
- 로딩: 없음 (즉시 처리)

### 6.4 반응형 디자인
- 데스크탑: 좌우 2열 레이아웃
- 모바일: 상하 2행 레이아웃

## 7. Technical Considerations

### 7.1 기술 스택
- **프레임워크**: Next.js 14+ (App Router)
- **UI 라이브러리**: Shadcn UI
- **스타일링**: Tailwind CSS
- **유틸리티**:
  - JSON 파싱: 네이티브 `JSON.parse()`
  - JWT 디코딩: 네이티브 `atob()` 또는 base64 라이브러리
  - SQL 파싱: 정규식 또는 간단한 문자열 처리

### 7.2 파일 구조
```
dev-utils/
├── app/
│   ├── layout.tsx
│   ├── page.tsx
│   └── globals.css
├── components/
│   ├── tools/
│   │   ├── JsonFormatter.tsx
│   │   ├── JwtDecoder.tsx
│   │   └── SqlBinder.tsx
│   └── ui/ (Shadcn components)
├── lib/
│   ├── json-handler.ts
│   ├── jwt-handler.ts
│   └── sql-handler.ts
└── package.json
```

### 7.3 클라이언트 사이드 처리
- `"use client"` 지시어로 모든 도구를 Client Component로 구현
- 데이터가 서버로 전송되지 않음

### 7.4 에러 처리
- 각 도구별로 시도-실패 방식의 검증
- 사용자 입력값 길이 제한 (예: 100KB 이상 입력 거부)
- 구체적인 에러 메시지로 사용자 가이드

### 7.5 성능 고려사항
- 로컬 변환이므로 지연 없음
- 대용량 JSON (10MB+)은 처리 불가 표시

## 8. Success Metrics

### 단기 (1개월)
- ✅ 3가지 도구(JSON, JWT, SQL) 완성 및 배포
- ✅ 팀 사용성 테스트 완료
- ✅ 사용자 피드백 수집 (최소 5명)

### 중기 (3개월)
- 📈 팀 내 월 사용량 100회 이상 (추적 필요시)
- 📚 3개 이상의 추가 도구 추가
- 🔄 사용자 피드백을 통한 개선 2회 이상

### 장기 (6개월)
- 🎯 5개 카테고리 모든 도구 구현 완료
- 💾 선택사항: 팀 내 도구 표준화 달성
- 🌟 사내 개발 문화에 정착
